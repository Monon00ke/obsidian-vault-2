## 1. Файловые дескрипторы

### Что такое файловый дескриптор

**Файловый дескриптор** — целое число, которое в контексте конкретного процесса однозначно идентифицирует открытый «файл» (в широком смысле: реальный файл, устройство, pipe, сокет и т.д.).

Внутри ядра с каждым процессом связана таблица открытых файлов — массив структур. Файловый дескриптор — это **индекс** в этом массиве. Ядро хранит по каждому индексу информацию о том, что именно открыто и в каком состоянии (например, текущая позиция для файла).

### Стандартные дескрипторы

Три дескриптора открыты в каждом процессе с самого начала:

| Дескриптор | Имя | Назначение |
|---|---|---|
| `0` | stdin | Стандартный ввод |
| `1` | stdout | Стандартный вывод |
| `2` | stderr | Стандартный вывод ошибок |

### Работа с дескрипторами в Python

```python
import os

# Открыть файл через os — получить числовой дескриптор
fd = os.open('file.txt', os.O_RDONLY)
data = os.read(fd, 5)   # читать 5 байт через системный вызов read
os.close(fd)

# Получить дескриптор из высокоуровневого файлового объекта
f = open('file.txt', 'r')
print(f.fileno())   # например, 3

# Создать файловый объект из числового дескриптора
f = os.fdopen(fd, 'r')
```

### Лимит дескрипторов и утечки

По умолчанию один процесс может держать открытыми не более **1024** файлов одновременно (лимит настраивается). При превышении системный вызов вернёт ошибку `Too many open files`.

**Причины исчерпания лимита:**
1. **Небрежный код** — файлы открываются и не закрываются. Это утечка дескрипторов.
2. **Объективная потребность** — высоконагруженные приложения (СУБД, компиляторы) действительно держат открытыми сотни файлов одновременно.

**Пример утечки:**
```python
files = []
for i in range(2000):
    files.append(open('file.txt'))   # ссылки хранятся → GC не срабатывает
    # OSError: [Errno 24] Too many open files
```

**Правильный вариант:** всегда закрывать файлы явно.
```python
for i in range(2000):
    f = open('file.txt')
    # ... работа с f ...
    f.close()
```

### Сборщик мусора Python и дескрипторы

В CPython файловый объект закрывается автоматически при уничтожении (через `__del__`). Это может создать иллюзию, что закрывать вручную не нужно. Полагаться на это **не следует**:

- Это особенность конкретной реализации CPython, другие реализации могут работать иначе.
- Разные политики GC могут откладывать освобождение объектов.
- Файловые ресурсы не должны зависеть от дисциплины управления памятью.

**Лучшая практика** — использовать контекстный менеджер:
```python
with open('file.txt') as f:
    data = f.read()
# f.close() вызовется автоматически при выходе из блока with
```

## 2. Системный вызов `pipe`

### Что создаёт `pipe`

```python
r_fd, w_fd = os.pipe()
```

`os.pipe()` делает системный вызов `pipe`, который:
1. Выделяет в памяти ядра буфер размером ~64 КБ.
2. Создаёт два файловых дескриптора для работы с этим буфером:
   - `r_fd` — для **чтения** из буфера
   - `w_fd` — для **записи** в буфер

Pipe реализует принцип «всё есть файл»: буфер в памяти доступен через те же интерфейсы `read`/`write`, что и обычные файлы.

### Запись и чтение

```python
import os

r_fd, w_fd = os.pipe()

# Запись: данные должны быть байтами
os.write(w_fd, "Hello, pipe!".encode())

# Чтение
data = os.read(r_fd, 1024)
print(data.decode())   # Hello, pipe!

os.close(r_fd)
os.close(w_fd)
```

### Блокировка при заполнении

Если буфер pipe заполнен (~64 КБ), вызов `os.write()` **заблокируется** до тех пор, пока другой процесс не прочитает данные из другого конца. В однопроцессном коде это приведёт к дедлоку.

```python
# ОПАСНО: попытка записать больше 64 КБ без читающего процесса
os.write(w_fd, b'x' * (64 * 1024 + 1))  # зависнет!
```

### Просмотр дескрипторов через /proc

Файловые дескрипторы текущего процесса можно увидеть в `/proc`:

```bash
ls -la /proc/$(python3 -c "import os; print(os.getpid())")/fd/
```

## 3. Pipe для межпроцессного взаимодействия

### Наследование дескрипторов при fork

При `fork()` дочерний процесс **наследует все открытые дескрипторы** родителя. Это ключевое свойство, позволяющее использовать pipe для IPC.

**Важный нюанс:** наследованные дескрипторы ссылаются на **одни и те же файловые объекты** в ядре (включая общую позицию чтения). Для обычных файлов это может вызывать нежелательное поведение. Для pipe это именно то, что нужно.

### Соглашение по закрытию ненужных концов

После `fork` каждый процесс должен **закрыть** тот конец pipe, который ему не нужен:

```python
r_fd, w_fd = os.pipe()
pid = os.fork()

if pid == 0:
    # Дочерний: только пишет
    os.close(r_fd)   # закрыть ненужный конец для чтения
    os.write(w_fd, b"hello from child")
    os.close(w_fd)
    os._exit(0)
else:
    # Родитель: только читает
    os.close(w_fd)   # закрыть ненужный конец для записи
    data = os.read(r_fd, 1024)
    print("Received:", data.decode())
    os.close(r_fd)
    os.wait()
```

**Почему важно закрывать ненужные концы:**
- Если w_fd открыт у родителя, а ребёнок записал и закрыл свой w_fd — родитель при чтении получит EOF только когда закроет и свой w_fd. Пока хоть один процесс держит w_fd открытым, `read()` из r_fd будет блокироваться в ожидании данных.

### Шаблон 1: Синхронизация «ребёнок готов»

```python
import os, time

r_fd, w_fd = os.pipe()
pid = os.fork()

if pid == 0:
    os.close(r_fd)
    # Инициализация (имитируем задержкой)
    time.sleep(2)
    print("Child: initialized")
    os.write(w_fd, b"ready")   # сигнал готовности
    os.close(w_fd)
    os._exit(0)
else:
    os.close(w_fd)
    print("Parent: waiting for child to initialize...")
    os.read(r_fd, 5)   # блокируется до получения сигнала
    print("Parent: child is ready, proceeding")
    os.close(r_fd)
    os.wait()
```

Pipe здесь используется как легковесный механизм синхронизации: родитель ждёт события в дочернем процессе без polling.

### Шаблон 2: Двусторонний канал родитель ↔ ребёнок

Для двустороннего взаимодействия нужны **два** pipe:

```python
# pipe_parent_to_child: родитель пишет, ребёнок читает
p2c_r, p2c_w = os.pipe()
# pipe_child_to_parent: ребёнок пишет, родитель читает
c2p_r, c2p_w = os.pipe()

pid = os.fork()

if pid == 0:
    os.close(p2c_w)   # ребёнок не пишет в p2c
    os.close(c2p_r)   # ребёнок не читает из c2p
    request = os.read(p2c_r, 1024)
    os.write(c2p_w, b"Response: " + request)
    os.close(p2c_r)
    os.close(c2p_w)
    os._exit(0)
else:
    os.close(p2c_r)   # родитель не читает из p2c
    os.close(c2p_w)   # родитель не пишет в c2p
    os.write(p2c_w, b"Hello child")
    response = os.read(c2p_r, 1024)
    print(response.decode())
    os.close(p2c_w)
    os.close(c2p_r)
    os.wait()
```

## 4. Построение конвейера (pipeline)

### Идея

Команда вида `ls | grep txt | sed 's/.txt//'` требует создать три процесса, соединённых двумя pipe:

```
[ls] --pipe0--> [grep] --pipe1--> [sed]
```

Каждый процесс читает из своего pipe (stdin → дескриптор 0) и пишет в следующий pipe (stdout → дескриптор 1).

### Алгоритм

1. Разбить командную строку по символу `|` → получить список команд.
2. Создать N-1 pipe для N команд.
3. Для каждой команды: `fork()` + настроить дескрипторы + `exec()`.
4. Родитель закрывает все pipe и ждёт завершения всех детей.

### Схема распределения дескрипторов

Для конвейера `cmd0 | cmd1 | cmd2`:

| Процесс | stdin (читает из) | stdout (пишет в) |
|---|---|---|
| cmd0 (i=0) | stdin терминала | w_fd pipe[0] |
| cmd1 (i=1) | r_fd pipe[0] | w_fd pipe[1] |
| cmd2 (i=2) | r_fd pipe[1] | stdout терминала |

### Проблема переназначения дескрипторов

После fork у дочернего процесса pipe-дескрипторы имеют произвольные номера (например, 3, 4, 5, 6). Но процессы ожидают читать из дескриптора `0` (stdin) и писать в дескриптор `1` (stdout).

Решение: системный вызов **`dup2`** — подробно на следующем занятии.

```python
# dup2 переназначит дескриптор pipe на стандартный вход/выход
os.dup2(pipe_r_fd, 0)   # pipe_r_fd становится stdin (дескриптор 0)
os.dup2(pipe_w_fd, 1)   # pipe_w_fd становится stdout (дескриптор 1)
```

### Набросок реализации конвейера

```python
import os, sys

# Команды для конвейера
cmds = [
    ['/usr/bin/ls', 'ls', '-l'],
    ['/usr/bin/grep', 'grep', 'py'],
    ['/usr/bin/sed', 'sed', 's/.py//'],
]

# Создать N-1 pipe
pipes = [os.pipe() for _ in range(len(cmds) - 1)]

pids = []
for i, cmd in enumerate(cmds):
    pid = os.fork()
    if pid == 0:
        # Дочерний: закрыть ненужные концы pipe
        for j, (r, w) in enumerate(pipes):
            if j != i - 1:    # не наш входной pipe
                os.close(r)
            if j != i:        # не наш выходной pipe
                os.close(w)

        # Переназначить stdin/stdout (будет разобрано на следующем занятии)
        if i > 0:
            os.dup2(pipes[i-1][0], 0)   # stdin = предыдущий pipe
        if i < len(cmds) - 1:
            os.dup2(pipes[i][1], 1)     # stdout = следующий pipe

        os.execv(cmd[0], cmd[1:])
        os._exit(1)

    pids.append(pid)

# Родитель закрывает все pipe и ждёт
for r, w in pipes:
    os.close(r)
    os.close(w)

for pid in pids:
    os.waitpid(pid, 0)
```

## 5. Обзор механизмов IPC в Unix

| Механизм | Описание | Типичное применение |
|---|---|---|
| **Pipe** | Однонаправленный буфер в памяти | Конвейеры, синхронизация родственных процессов |
| **FIFO (named pipe)** | Pipe с именем в файловой системе | IPC между произвольными процессами |
| **Разделяемая память** | Общие страницы памяти | Максимальная производительность, нужна синхронизация |
| **Семафор** | Счётчик для ограничения параллельного доступа | Ограничение числа одновременных операций |
| **Очередь сообщений** | Структурированная асинхронная передача | Слабосвязанные компоненты |
| **Сигналы** | Асинхронное уведомление процесса | Управление жизненным циклом (SIGTERM, SIGKILL и др.) |
| **Unix-сокет** | Двунаправленное соединение | IPC на одной машине с сетевым API |
| **TCP/UDP** | Сетевые сокеты | Взаимодействие между машинами |

## Итог лекции

Разобраны:
- **Файловые дескрипторы**: природа (индекс в таблице ядра), стандартные 0/1/2, получение из Python-объекта (`fileno()`), обратное (`fdopen`);
- **Лимит 1024** и утечки: две причины, как диагностировать, как закрывать явно; роль GC в CPython и почему на него нельзя полагаться; `/proc/<pid>/fd` для наблюдения;
- **Системный вызов `pipe`**: что создаёт (буфер 64 КБ + два дескриптора), чтение/запись через `os.read`/`os.write`, блокировка при заполнении;
- **Наследование дескрипторов при fork**: все дескрипторы наследуются ребёнком; общая позиция чтения; правило закрытия ненужных концов pipe;
- **Шаблоны использования pipe**: синхронизация «ребёнок готов»; двусторонний канал через два pipe;
- **Построение конвейера**: алгоритм, схема дескрипторов, проблема переназначения stdin/stdout;
- **`dup2`** — анонс темы следующего занятия.

На следующем занятии: `dup2`, полная реализация конвейера, сигналы.
