## 1. FIFO (Named Pipe)

### Что такое FIFO

**FIFO** (First In, First Out) — именованный канал. Это тот же pipe, но с именем в файловой системе. Благодаря имени его могут открыть произвольные процессы, не состоящие в родственных отношениях.

| | Pipe | FIFO |
|---|---|---|
| Наличие имени в ФС | Нет | Есть |
| Доступен произвольным процессам | Нет (только родственным) | Да |
| Хранит данные на диске | Нет (только в памяти) | Нет (только в памяти) |
| Семантика | Байтовый поток | Байтовый поток |

### Создание и использование FIFO

```bash
# Создать FIFO
mkfifo /tmp/my_fifo

# В одном терминале — читатель
cat /tmp/my_fifo

# В другом терминале — писатель
echo "hello" > /tmp/my_fifo
```

Попытка открыть FIFO на чтение блокируется, пока не появится писатель — и наоборот. Если писатель пишет быстрее, чем читатель читает, запись приостанавливается — как и в обычном pipe. Несколько читателей могут параллельно вычитывать данные из одного FIFO, помогая справляться с нагрузкой.

### Ограничения

FIFO — байтовый поток без границ сообщений. При наличии нескольких читателей нельзя гарантировать, что каждый из них получит целое сообщение: читатели могут «разрезать» данные в произвольных местах. Для структурированных сообщений используйте очереди сообщений.

## 2. Сигналы

### Что такое сигнал

**Сигнал** — асинхронное уведомление, отправляемое процессу. Это ограниченный способ межпроцессного взаимодействия: передаётся только номер сигнала, никаких данных.

Источники сигналов:
- Другой процесс (через системный вызов `kill`)
- Ядро ОС (например, при выходе за пределы допустимой памяти — `SIGSEGV`)
- Пользователь с терминала (`Ctrl+C` → `SIGINT`, `Ctrl+Z` → `SIGTSTP`)

### Основные сигналы

| Номер | Имя | Действие по умолчанию | Описание |
|---|---|---|---|
| 1 | `SIGHUP` | Завершение | Разрыв соединения с терминалом |
| 2 | `SIGINT` | Завершение | Прерывание (Ctrl+C) |
| 3 | `SIGQUIT` | Core dump | Выход с дампом (Ctrl+\\) |
| 9 | `SIGKILL` | Завершение | Принудительное уничтожение (нельзя игнорировать) |
| 11 | `SIGSEGV` | Core dump | Недопустимый доступ к памяти |
| 15 | `SIGTERM` | Завершение | Вежливая просьба завершиться |
| 17 | `SIGCHLD` | Игнорирование | Дочерний процесс завершился |
| 18 | `SIGCONT` | Продолжение | Возобновить остановленный процесс |
| 19 | `SIGSTOP` | Остановка | Приостановить процесс (нельзя игнорировать) |

Посмотреть все сигналы: `kill -l`

### Три варианта реакции на сигнал

1. **Default** — действие по умолчанию для данного сигнала (завершение, игнорирование или core dump).
2. **Ignore** — сигнал обрабатывается, но никакого действия не производится; бумажка выбрасывается.
3. **Handler** — вызывается пользовательская функция-обработчик.

**SIGKILL и SIGSTOP нельзя перехватить, заблокировать или проигнорировать.** Это гарантирует, что система всегда может управлять жизненным циклом любого процесса.

### Отправка сигналов

```bash
# Послать SIGTERM (по умолчанию)
kill <PID>

# Послать конкретный сигнал по номеру
kill -9 <PID>      # SIGKILL
kill -19 <PID>     # SIGSTOP
kill -18 <PID>     # SIGCONT

# Послать по имени
kill -SIGTERM <PID>
```

```python
import os, signal

os.kill(pid, signal.SIGTERM)
os.kill(pid, signal.SIGKILL)
```

### SIGTERM vs SIGKILL: правильный порядок завершения

| | SIGTERM (15) | SIGKILL (9) |
|---|---|---|
| Можно перехватить | Да | Нет |
| Graceful shutdown | Возможен | Невозможен |
| Когда использовать | Сначала | Если SIGTERM не помог |

**Рекомендуемая практика:** сначала отправить `SIGTERM` и подождать несколько секунд. Если процесс не завершился — тогда `SIGKILL`.

## 3. Маскирование и игнорирование сигналов

### Ключевое различие

**Маскирование** — временное откладывание обработки сигнала. Факт прихода сигнала сохраняется. Когда маска будет снята, отложенный сигнал будет обработан.

**Игнорирование** — форма обработки: сигнал принят и немедленно выброшен.

Аналогия со стаканчиком и бумажками:
- **Маскирование**: «я сейчас не смотрю в стаканчик; бумажка там лежит и ждёт»
- **Игнорирование**: «я взял бумажку из стаканчика и выбросил в мусор»

### Маскирование в Python

```python
import signal

# Посмотреть текущую маску (пустая у нового процесса)
current_mask = signal.pthread_sigmask(signal.SIG_BLOCK, [])
print(current_mask)   # set()

# Замаскировать SIGTERM
signal.pthread_sigmask(signal.SIG_BLOCK, {signal.SIGTERM})

# Снять маску (сигнал будет обработан, если приходил пока был замаскирован)
signal.pthread_sigmask(signal.SIG_UNBLOCK, {signal.SIGTERM})

# Установить конкретную маску (заменить текущую)
signal.pthread_sigmask(signal.SIG_SETMASK, {signal.SIGINT, signal.SIGTERM})
```

### Установка обработчика

```python
import signal, os

def handler(signum, frame):
    print(f"Received signal {signum}, graceful shutdown...")
    # ... завершающие действия ...
    os._exit(0)

# Установить обработчик
signal.signal(signal.SIGTERM, handler)

# Игнорировать сигнал
signal.signal(signal.SIGHUP, signal.SIG_IGN)

# Вернуть обработчик по умолчанию
signal.signal(signal.SIGTERM, signal.SIG_DFL)
```

## 4. Когда и как доставляется сигнал

### Момент обработки

Сигнал обрабатывается в момент **перехода с уровня ядра на уровень приложения**. Это может произойти:
- При возврате из прерывания (например, тикнул таймер планировщика)
- При выходе из системного вызова

Пока выполняется код приложения (пользовательский режим), ядро не может вмешаться напрямую — нет механизма «прервать прямо здесь и сейчас». Поэтому обработка может быть слегка отложена (единицы микросекунд для человека незаметны, но значимы на уровне процессорных инструкций).

### Сигнал во время системного вызова

Если сигнал приходит, когда процесс **спит внутри системного вызова** (например, `wait`, `read`, блокирующий `pipe`):

- Обработчик сигнала **не вызывается изнутри кода ядра** — это могло бы нарушить инварианты ядра.
- Вместо этого системный вызов **принудительно завершается** с кодом ошибки `EINTR` («прерван сигналом»).
- Приложение должно обработать `EINTR` и при необходимости повторить системный вызов.

```python
import os, errno

while True:
    try:
        pid, status = os.wait()
        break
    except OSError as e:
        if e.errno == errno.EINTR:
            continue   # был прерван сигналом, повторить
        raise
```

## 5. Теория гарантий доставки

При проектировании любой коммуникационной системы возникает выбор между тремя уровнями гарантий:

| Гарантия | Описание | Пример |
|---|---|---|
| **Exactly once** | Доставляется ровно один раз | Идеал; сложно реализовать |
| **At most once** | Доставляется не более одного раза (может потеряться) | Сигналы в Unix |
| **At least once** | Доставляется хотя бы один раз (может дублироваться) | TCP с наивной логикой повторов |

### Почему сигналы используют «at most once»

Реализация «exactly once» требует подтверждений от получателя. Но в асинхронной системе подтверждение тоже может потеряться — тогда отправитель не знает, дошло ли сообщение, и может отправить повторно → получим «at least once» с дубликатами.

Для сигналов выбрано «at most once»: простая реализация через один бит на сигнал в структуре процесса. Плата — возможная потеря сигналов при быстрой посылке одного и того же сигнала.

**Практический вывод:** не используйте сигналы для счётчиков или для задач, где важна гарантия обработки каждого события. Для этого подходят очереди сообщений или другие механизмы IPC.

### Синхронная vs асинхронная коммуникация

| | Синхронная | Асинхронная |
|---|---|---|
| Предсказуемость | Высокая | Ниже |
| Риск заторов | Есть (блокировка в ожидании) | Нет |
| Пропускная способность | Ниже | Выше |
| Гарантии доставки | Проще обеспечить | Сложнее |
| Пример | `wait()`, блокирующий `read()` | Сигналы, UDP |

## 6. Практические сценарии использования сигналов

### Graceful shutdown (SIGTERM)

```python
import signal, sys

def graceful_shutdown(signum, frame):
    print("Shutting down gracefully...")
    # закрыть соединения, сохранить состояние
    sys.exit(0)

signal.signal(signal.SIGTERM, graceful_shutdown)
signal.signal(signal.SIGINT, graceful_shutdown)   # Ctrl+C

# основной цикл работы
while True:
    do_work()
```

### Reload конфигурации по SIGHUP

```python
import signal

config = load_config()

def reload_config(signum, frame):
    global config
    config = load_config()
    print("Config reloaded")

signal.signal(signal.SIGHUP, reload_config)
```

### Управление процессом из командной строки

```bash
# Остановить процесс
kill -SIGSTOP <PID>

# Возобновить процесс
kill -SIGCONT <PID>

# Попросить завершиться
kill -SIGTERM <PID>

# Принудительно уничтожить
kill -SIGKILL <PID>

# В bash: перевести текущий процесс в фон
Ctrl+Z    # SIGTSTP → suspended
bg        # SIGCONT → продолжить в фоне
fg        # перевести обратно на передний план
```

Разобраны:
- **FIFO**: именованный pipe; создание через `mkfifo`; доступен произвольным процессам; блокировка без читателя/писателя; ограничения байтового потока;

- **Момент доставки**: переход ядро→приложение; `EINTR` при прерывании системного вызова;
- **Теория гарантий доставки**: exactly once / at most once / at least once; почему сигналы используют «at most once»; синхронная vs асинхронная коммуникация.
