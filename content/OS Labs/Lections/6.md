# Лекция: sudo, файловые системы и монтирование

---

## 1. sudo и управление привилегиями

### Проблема с прямым входом под root

Классический способ получить права администратора — войти под пользователем **root** через `su` (superuser). Однако у этого подхода есть существенные недостатки:

- Если пароль root знают несколько человек, при смене роли сотрудника нужно менять пароль для всех.
- Когда человек работает в сеансе root, в логах все действия фиксируются как действия root — невозможно отследить, кто именно что сделал.

### sudo как альтернатива

**sudo** (superuser do) позволяет выполнять отдельные команды с правами root, не входя в его сеанс.

```bash
sudo mkdir /root/test_dir     # создать каталог в /root от имени root
sudo apt install пакет        # установить пакет
```

Ключевые отличия от `su`:

| | `su` | `sudo` |
|---|---|---|
| Вход в сеанс | Полный переход в root-сеанс | Разовое выполнение команды |
| Аутентификация | Пароль root | Пароль текущего пользователя |
| Отзыв прав | Нужно менять пароль root | Удалить пользователя из sudoers |
| Аудит | Кто именно действовал — неизвестно | В логах виден реальный пользователь |

### Файл /etc/sudoers

Список пользователей, имеющих право использовать `sudo`, хранится в файле **`/etc/sudoers`**. Этот файл нельзя изменить без прав root, а редактировать его принято только через команду `visudo` (она проверяет корректность синтаксиса).

### Real User и Effective User

Когда процесс выполняется через `sudo`, у него есть два понятия пользователя:

- **Real User (UID)** — реальный пользователь, от имени которого запущен процесс (например, `vasya`).
- **Effective User (EUID)** — пользователь, чьи права используются при проверке разрешений (например, `root`).

Это позволяет выполнить действие от имени root, но при этом сохранить в логах информацию о том, кто именно это сделал.

## 2. Концепция «всё есть файл»

Unix-системы построены на принципе: **всё является файлом**. Файл — это абстракция, из которой можно читать и в которую можно писать данные последовательно байт за байтом.

Примеры того, что является файлом:

| Объект | Описание |
|---|---|
| Обычный файл | Документ, исходный код, изображение |
| Каталог | Тоже файл — содержит список имён и ссылок на inode |
| stdin / stdout / stderr | Файловые дескрипторы 0, 1, 2 |
| `/dev/null` | Псевдофайл — «чёрная дыра», всё записанное исчезает |
| `/dev/zero` | Псевдофайл — при чтении отдаёт бесконечный поток нулей |
| `/dev/urandom` | Псевдофайл — при чтении отдаёт случайные байты |
| Файл устройства | Интерфейс к физическому или виртуальному устройству |
| Сокет, pipe | Тоже файлы |

### Файлы устройств в `/dev`

Каждое устройство (диск, сетевой интерфейс, терминал) представлено файлом в `/dev`. Запись в такой файл — это запись непосредственно в устройство. Чтение — чтение из устройства.

```bash
# Создать файл размером 1 МБ, заполненный нулями
head -c 1M /dev/zero > file.img

# Посмотреть содержимое диска в виде hex-дампа
od -A x -t x1z /dev/sda | head -20
```

## 3. Файловые системы: два смысла термина

Слово «файловая система» используется в двух разных значениях:

**Значение 1 — Виртуальная файловая система (VFS, Virtual File System):**
Абстрактное единое дерево каталогов, которое видит пользователь при работе с Linux. Это то, что отображается при `ls /`. VFS существует в памяти работающего ядра и может состоять из нескольких физических файловых систем.

**Значение 2 — Физическая файловая система:**
Структура данных, записанная на конкретном физическом носителе (диске, флешке). Определяет, как именно байты на диске интерпретируются как файлы и каталоги.

### Популярные физические файловые системы

| Файловая система | Где используется |
|---|---|
| **ext4** | Стандартная для Linux |
| **XFS** | Эффективна при большом количестве файлов в одном каталоге |
| **Btrfs** (Butter FS) | Современная, с поддержкой снапшотов |
| **FAT32** | Устаревшая Windows, флешки |
| **NTFS** | Современная Windows |
| **APFS** | macOS |

Linux поддерживает монтирование файловых систем Windows (FAT32, NTFS) через штатные механизмы. В обратную сторону Windows не умеет читать ext4 «из коробки».

В Linux можно создать собственную файловую систему и интегрировать её в экосистему — никаких специальных лицензий для этого не нужно.

## 4. Монтирование

### Принцип монтирования

**Монтирование** — это операция, которая привязывает физическую файловую систему к определённой точке виртуальной файловой системы (точке монтирования).

```
Физический диск (ext4) → смонтирован → /mnt/disk2
Флешка (FAT32)         → смонтирована → /media/usb
```

После монтирования содержимое диска становится доступно через путь точки монтирования.

### Команды

```bash
# Смонтировать устройство /dev/sdb1 в каталог /mnt/disk2
mount /dev/sdb1 /mnt/disk2

# Отмонтировать
umount /mnt/disk2

# Показать все смонтированные файловые системы
mount

# Показать использование дискового пространства
df -h

# Показать информацию о конкретном каталоге
df -h /mnt/disk2
```

Автоматическое монтирование при старте системы прописывается в файле **`/etc/fstab`**.

### Практический пример: псевдодиск из файла

Поскольку устройства — это файлы, можно создать «псевдодиск» из обычного файла:

```bash
# Создать файл-образ размером 500 МБ, заполненный нулями
head -c 500M /dev/zero > pseudo.img

# Отформатировать образ под ext2
mkfs.ext2 pseudo.img

# Создать точку монтирования и смонтировать образ
mkdir /tmp/mnt_test
mount -o loop pseudo.img /tmp/mnt_test

# Работать с файловой системой как обычно
ls /tmp/mnt_test
cp файл /tmp/mnt_test/

# Посмотреть информацию
df -h /tmp/mnt_test

# Отмонтировать
umount /tmp/mnt_test
```

Это полезно для тестирования программ, работающих с файловыми системами, — не нужно покупать несколько физических дисков.

## 5. Виртуальные (псевдо) файловые системы

Идею монтирования можно обобщить: к VFS можно подключать не только физические файловые системы, но и любые древовидные структуры данных, живущие в памяти ядра.

### /proc — информация о процессах

`/proc` — это виртуальная файловая система, отображающая данные ядра о процессах через файловый интерфейс.

```bash
ls /proc            # числа = PID запущенных процессов, плюс системные файлы

ls /proc/949/       # информация о процессе с PID 949
ls /proc/949/fd/    # файлы, открытые этим процессом (по дескриптору)
cat /proc/949/cmdline  # командная строка, которой запущен процесс
cat /proc/meminfo   # информация о памяти системы
cat /proc/cpuinfo   # информация о процессоре
```

Содержимое `/proc` не хранится на диске — оно генерируется ядром «на лету» при каждом чтении.

### /sys — параметры ядра и устройств

Аналогично `/proc`, файловая система `/sys` предоставляет доступ к параметрам ядра, настройкам устройств и драйверов.

### Проект FUSE

**FUSE** (Filesystem in Userspace) позволяет создавать собственные файловые системы без прав root и монтировать их в своём домашнем каталоге. Примеры применений: отображение базы данных через файловый интерфейс, просмотр облачного хранилища как локального каталога.

## 6. Ссылки на файлы: hardlink и symlink

### inode

**inode** (index node) — структура данных, хранящая метаданные файла: права доступа, владелец, размер, временны́е метки, и адреса блоков данных на диске. Имя файла хранится не в inode, а в записи каталога, которая указывает на inode.

### Символическая ссылка (symlink / soft link)

Создаётся командой `ln -s`. Это отдельный файл особого типа, содержимое которого — путь к целевому файлу.

```bash
ln -s /path/to/original link_name

# Пример:
ln -s /etc/hosts hosts_link
ls -l hosts_link     # hosts_link -> /etc/hosts
```

Особенности symlink:
- Если целевой файл удалён, symlink «зависает» (dangling link) — ссылка есть, но открыть нельзя.
- Может ссылаться на файл в другой файловой системе.
- Может ссылаться на каталог.

### Жёсткая ссылка (hardlink)

Создаётся командой `ln` (без `-s`). Добавляет новое имя в каталоге, указывающее на тот же inode.

```bash
ln original_file hard_link_name

# Проверить количество жёстких ссылок (2-я колонка ls -l)
ls -l
# -rw-r--r-- 2 user group 123 May 29 file     # 2 = две жёсткие ссылки
```

Особенности hardlink:
- Файл существует, пока существует хотя бы одна жёсткая ссылка (счётчик ссылок > 0).
- Не может ссылаться на каталог (во избежание циклов).
- Не может пересекать границы файловых систем (оба имени должны быть в одной ФС).
- Удаление одного имени не удаляет данные — только уменьшает счётчик ссылок.

### Сравнение

| | Symlink | Hardlink |
|---|---|---|
| Тип файла | Отдельный (lrwxrwxrwx) | То же, что оригинал |
| Собственный inode | Да | Нет (общий с оригиналом) |
| Разные ФС | Да | Нет |
| На каталоги | Да | Нет |
| При удалении цели | Становится «битой» | Данные сохраняются |

## 7. Типы файлов в Linux

В выводе `ls -l` тип файла определяется первым символом:

| Символ | Тип |
|---|---|
| `-` | Обычный файл |
| `d` | Каталог |
| `l` | Символическая ссылка |
| `c` | Символьное устройство (character device) |
| `b` | Блочное устройство (block device, например диск) |
| `p` | Именованный канал (pipe) |
| `s` | Сокет |

## Итог лекции

Разобраны:
- **sudo**: effective user vs real user, файл `/etc/sudoers`, преимущества перед прямым входом под root;
- **«всё есть файл»**: файлы устройств в `/dev`, псевдофайлы `/dev/null`, `/dev/zero`, `/dev/urandom`;
- **файловые системы**: два значения термина (VFS и физическая), популярные ФС, открытость экосистемы Linux;
- **монтирование**: привязка физической ФС к точке VFS, команды `mount`/`umount`/`df`, псевдодиск из файла;
- **виртуальные ФС**: `/proc` и `/sys` как примеры древовидных структур данных, FUSE;
- **ссылки**: inode, symlink vs hardlink, счётчик ссылок;
- **типы файлов**: `ls -l` и первый символ в правах.

На следующем занятии: **bash-программирование** (скрипты, переменные, условия, циклы).
