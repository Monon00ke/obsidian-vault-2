# Лекция: Контейнерная виртуализация — Namespaces, cgroups, UnionFS и Docker

---

## 1. Три механизма контейнерной виртуализации

Контейнерная виртуализация в Linux строится на трёх ключевых механизмах ядра:

| Механизм | Что делает |
|---|---|
| **Namespaces** | Изолирует глобальные ресурсы ОС для конкретного процесса |
| **Control groups (cgroups)** | Ограничивает потребление ресурсов (CPU, RAM и др.) |
| **UnionFS** | Объединяет несколько файловых систем в одну с послойным поиском |

Все три механизма — исключительная особенность Linux. В Windows их нет в нативном виде (только через WSL). В macOS — нет (Docker там использует скрытую Linux-виртуалку). Ближайший аналог есть только в FreeBSD (механизм **jail**).

## 2. Namespaces (пространства имён)

### Идея

**Namespace** — это механизм, позволяющий изменить «представление процесса о мире». Некоторые свойства системы глобальны (имя хоста, дерево процессов, список сетевых интерфейсов), но с помощью namespace каждый процесс может видеть свою изолированную копию этих свойств.

По умолчанию все процессы работают в одном общем namespace — «коммуналке». Namespace позволяет отселить процесс в собственное изолированное пространство.

### 8 типов namespaces в Linux

| Тип | Что изолирует |
|---|---|
| **UTS** | Имя хоста (`hostname`) и доменное имя |
| **PID** | Дерево процессов и их идентификаторы |
| **Mount** | Представление виртуальной файловой системы (VFS) |
| **Network** | Сетевые интерфейсы, таблицы маршрутизации, порты |
| **User** | Пользователи и группы |
| **IPC** | Механизмы IPC: семафоры, shared memory, очереди сообщений |
| **Time** | Часовые настройки и системное время |
| **Cgroup** | Видимость иерархии cgroups |

### Системные вызовы для работы с namespaces

**`unshare`** — отселить текущий процесс в новый namespace:

```bash
# Создать UTS-namespace и запустить bash в нём
sudo unshare --uts bash

# Теперь внутри этого bash меняем hostname — это не затронет остальную систему
hostname my-sandbox
hostname   # my-sandbox

# В другом терминале hostname не изменился
```

**`clone`** — создать новый процесс сразу в новом namespace (передаётся функция нового процесса + флаги namespace):

```bash
# Команда unshare в bash — это обёртка над системным вызовом unshare
man 2 unshare
man 2 clone
```

### PID namespace в Python

```python
import os

# Перед unshare
print(f"Before: PID={os.getpid()}, PPID={os.getppid()}")

# os.unshare доступен в Python 3.12+
# os.unshare(os.CLONE_NEWPID)  # создать новый PID namespace

pid = os.fork()
if pid == 0:
    # Дочерний процесс в новом PID namespace видит себя как PID=1
    print(f"Child in namespace: PID={os.getpid()}, PPID={os.getppid()}")
    # PID=1, PPID=0 — считает себя корневым процессом
    os._exit(0)
else:
    print(f"Parent: sees child as PID={pid}")
    os.wait()
```

Вывод (при использовании PID namespace):
```
Parent PID=12349, PPID=12348
Child in namespace: PID=1, PPID=0
Parent sees child as PID=12350
```

Ключевое свойство: изоляция **односторонняя** — хост видит все процессы контейнера, но процессы внутри контейнера не видят процессы хоста.

## 3. Control groups (cgroups)

### Что такое cgroups

**Control groups** — механизм ядра для управления потреблением ресурсов группой процессов. Каждый процесс принадлежит cgroup, для которой можно выставить лимиты.

### Ограничиваемые ресурсы

| Ресурс | Пример лимита |
|---|---|
| **Память** | Не более 1 ГБ RAM |
| **CPU** | Не более 20% процессорного времени |
| **Дисковый I/O** | Ограничение пропускной способности |
| **Сеть** | Ограничение трафика |

### Применение

**Отладка производительности:** ограничить контейнер по памяти, чтобы убедиться, что алгоритм вписывается в заданный бюджет.

**Production-системы:** в дата-центре одна физическая машина с 64 ГБ RAM может одновременно запускать десятки контейнеров, каждый с лимитом, например, 4 ГБ. Cgroups гарантируют, что ни один контейнер не выйдет за рамки выделенного.

### Просмотр и управление cgroups

```bash
# Просмотр иерархии cgroups
ls /sys/fs/cgroup/

# Утилита для просмотра
systemd-cgls

# Ограничение памяти для процесса через cgexec (из пакета cgroup-tools)
cgexec -g memory:mygroup ./my_process
```

## 4. UnionFS — многослойная файловая система

### Проблема, которую решает UnionFS

При запуске 10 контейнеров на основе одного и того же базового образа Ubuntu 98% содержимого файловой системы одинаково. Было бы расточительно хранить 10 копий.

### Принцип работы

**UnionFS** объединяет несколько реальных файловых систем («слоёв») в одну виртуальную. При операции с файлом поиск идёт послойно — от верхнего (более позднего) к нижнему:

```
Слой 3 (самый верхний, последние изменения)  ← ищем здесь первым
Слой 2 (дополнения, например Python)
Слой 1 (базовый образ Ubuntu)                ← ищем здесь последним
```

- Если файл найден в более верхнем слое — используется он.
- Для листинга каталога проходятся все слои; файлы из верхних слоёв перекрывают одноимённые из нижних.

### Copy-on-Write

Базовые слои доступны **только для чтения**. При изменении файла из нижнего слоя происходит **Copy-on-Write**: файл копируется в верхний (записываемый) слой, где и редактируется. Оригинал в нижнем слое остаётся нетронутым.

## 5. Docker

### Docker как продуктовая реализация

**Docker** — удобная высокоуровневая оболочка над namespaces, cgroups и UnionFS. Предоставляет CLI, API, реестр образов и декларативный формат описания контейнеров.

### Основные понятия

| Понятие | Аналогия | Описание |
|---|---|---|
| **Image (образ)** | Исполняемый файл | Набор слоёв на диске; неизменяемый артефакт |
| **Container (контейнер)** | Запущенный процесс | Работающий образ; создаёт при необходимости записываемый слой |
| **Layer (слой)** | — | Дельта файловой системы; неизменяемый после создания |
| **Dockerfile** | — | Декларативный скрипт для сборки образа |
| **Docker Hub** | — | Публичный реестр образов |

**Соотношение образ/контейнер** аналогично бинарный файл/процесс: из одного образа можно запустить много контейнеров одновременно.

### Dockerfile: базовый синтаксис

```dockerfile
# Базовый образ (первый слой)
FROM ubuntu:24.04

# Обновить пакеты и установить Python
RUN apt-get update && apt-get install -y python3 python3-pip

# Установить зависимости приложения
RUN pip3 install numpy

# Скопировать файлы приложения в образ
COPY app.py /app/app.py

# Рабочий каталог
WORKDIR /app

# Команда по умолчанию при запуске контейнера
CMD ["python3", "app.py"]
```

Каждая инструкция `RUN`, `COPY`, `ADD` создаёт новый слой. `FROM`, `WORKDIR`, `CMD`, `ENV` — вспомогательные, не создают слоёв.

### Сборка образа

```bash
# Создать каталог для проекта
mkdir my-image && cd my-image

# Создать Dockerfile (см. пример выше)

# Собрать образ с тегом
docker build -t my-app:0.1 .

# Посмотреть список образов
docker image ls
docker image ls -a   # все, включая промежуточные слои
```

### Запуск контейнера

```bash
# Интерактивный запуск: войти в bash внутри контейнера
docker run -it ubuntu:24.04 bash

# Внутри контейнера — изолированная среда:
# ps aux покажет только bash + ps, PID=1 у bash
# hostname будет уникальным

# Выйти из контейнера
exit

# Запуск в фоне (detached)
docker run -d --name my-server my-app:0.1

# Запуск с ограничениями ресурсов (cgroups)
docker run -m 512m --cpus="0.5" ubuntu:24.04 bash
# -m 512m   — лимит памяти 512 МБ
# --cpus    — лимит CPU (0.5 = половина одного ядра)
```

### Управление контейнерами

```bash
# Список работающих контейнеров
docker ps
docker container ls

# Список всех контейнеров (включая остановленные)
docker ps -a

# Остановить контейнер
docker stop <container_id>

# Удалить контейнер
docker rm <container_id>

# Выполнить команду в работающем контейнере
docker exec -it <container_id> bash
```

### Сохранение изменений: commit

Если вы изменили что-то внутри работающего контейнера и хотите зафиксировать новый образ:

```bash
# Внутри контейнера:
apt-get install -y python3

# Из хост-системы — зафиксировать состояние контейнера как новый образ
docker container commit <container_id> my-app:0.2
```

**Рекомендуется** описывать изменения через Dockerfile, а не через commit — так образы воспроизводимы и понятны.

### Многослойность на практике

```
Image A: ubuntu:24.04 + Python  → слой 1 (Ubuntu) + слой 2 (Python)
Image B: ubuntu:24.04 + Go      → слой 1 (Ubuntu) + слой 3 (Go)

Слой 1 (Ubuntu) физически хранится один раз — общий для обоих образов.
Экономия: не нужно дублировать сотни МБ базового образа.
```

```bash
# Посмотреть слои конкретного образа
docker image history ubuntu:24.04
```

### Docker Hub и реестры образов

```bash
# Скачать готовый образ
docker pull ubuntu:24.04
docker pull python:3.12-slim

# Поиск образов
docker search nginx

# Загрузить свой образ в реестр
docker tag my-app:0.1 myusername/my-app:0.1
docker push myusername/my-app:0.1
```

## 6. Контейнеры vs. виртуальные машины

| | Контейнеры (Docker) | Виртуальные машины (VirtualBox, KVM) |
|---|---|---|
| Уровень изоляции | Namespace + cgroups (ядро Linux общее) | Полная виртуализация железа |
| Накладные расходы | Минимальные | Высокие (отдельное ядро ОС) |
| Запуск | Секунды | Минуты |
| Потребление RAM | Только процессы приложения | Процессы + ядро ОС виртуалки |
| Безопасность | Ниже (общее ядро) | Выше (полная изоляция) |
| Портируемость | Высокая | Средняя |
