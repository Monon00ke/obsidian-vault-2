## 1. Виртуальная память и своп (продолжение)

### Механизм вытеснения страниц на диск (swap)

Страничная организация памяти позволяет реализовать **своп**: страница памяти, давно не используемая, может быть выгружена на диск. Физическая страница при этом освобождается для других нужд, а запись в таблице страниц помечается как «не в памяти».

Когда процесс обращается к вытесненной странице:
1. Процессор обнаруживает, что страница отсутствует в памяти (page fault).
2. Возникает **trap (ловушка)** — аналог прерывания, но с перезапуском инструкции после обработки.
3. Ядро находит страницу на диске, загружает её в физическую память (вытесняя другую «холодную» страницу).
4. Таблица страниц обновляется. Инструкция исполняется повторно — теперь успешно.

**Отличие trap от interrupt:**

| | Прерывание (interrupt) | Ловушка (trap) |
|---|---|---|
| Причина | Асинхронное внешнее событие (таймер, устройство) | Синхронная ситуация при исполнении инструкции |
| Действие после обработки | Продолжить выполнение следующей инструкции | Повторить инструкцию, вызвавшую ловушку |

### Когда своп полезен, а когда вреден

**Своп полезен**, когда процессов много, но большинство из них не работают активно (спят, ждут событий). Сумма виртуальных адресных пространств может превышать физическую память.

**Своп вреден**, когда все процессы активно работают и постоянно обращаются к своим данным. В этом случае происходит **thrashing**: страницы непрерывно вытесняются и загружаются обратно, диск работает на полную, производительность резко падает.

На высоконагруженных серверах своп нередко **отключают намеренно**: лучше получить честный отказ памяти и разобраться с проблемой, чем незаметно деградировать.

## 2. Дополнительные возможности страничной организации

Помимо основных целей (гибкая адресация и изоляция), страничная организация позволяет:

| Возможность | Механизм |
|---|---|
| Упрощение управления памятью | Работа с блоками фиксированного размера (4 КБ) вместо произвольных диапазонов |
| Разделяемая память | Страница из одного адресного пространства отображается в другое |
| Права доступа к страницам | Биты R/W/X в записи таблицы страниц |
| Одна запись, много читателей | Один процесс имеет R/W, другие — только R |
| Своп | Страница помечается как отсутствующая; загружается по запросу |

## 3. Изоляция ядра от процессов

### Проблема

Процессы нуждаются в услугах ядра (открыть файл, создать процесс и т.д.). Для этого нужен доступ к структурам данных ядра. Но если процесс может напрямую читать и писать эти структуры — он может их повредить: случайно из-за бага или намеренно.

### Решение: режимы работы процессора

Современные процессоры имеют два основных режима работы:

| Режим | Кто работает | Возможности |
|---|---|---|
| **Privileged (Ring 0, kernel mode)** | Код ядра | Полный доступ к памяти, управление устройствами, смена таблиц страниц |
| **User mode (Ring 3)** | Код приложений | Ограниченный набор инструкций; обращение к структурам ядра запрещено |

Переключение из user mode в kernel mode происходит **только через системный вызов** — контролируемую точку входа в ядро.

### Real mode и protected mode

| Режим | Описание |
|---|---|
| **Real mode** | Процессор работает при включении; нет страничной организации, нет разделения ядро/приложение; вся память доступна всем |
| **Protected mode** | Активируется ядром после старта; включает страничную организацию, разделение привилегий |

При загрузке системы процессор стартует в real mode. Ядро настраивает структуры данных и переходит в protected mode. В protected mode создаётся первый процесс, который уже работает в user mode.

**Важно:** понятия «kernel mode / user mode» и «root / non-root» — разные уровни:
- **root / non-root** — разграничение прав внутри user mode (кто может делать привилегированные операции в рамках ОС).
- **kernel mode / user mode** — аппаратное разграничение: код ядра vs код любых процессов, включая запущенные под root.

## 4. Системные вызовы

### Что такое системный вызов

**Системный вызов** — параметризованная операция, исполняемая ядром по запросу процесса. Процесс исполняет специальную инструкцию (`syscall` на x86-64), передавая номер нужного вызова и аргументы. Процессор переключается в kernel mode, ядро выполняет операцию, возвращает результат, процессор возвращается в user mode.

Аналогия: учреждение с пропускным режимом. Посетитель не входит внутрь, но может воспользоваться фиксированным набором услуг через окошко регистратуры.

### Системный вызов vs. библиотечная функция

| | Системный вызов | Библиотечная функция |
|---|---|---|
| Пример | `write`, `read`, `open`, `fork` | `strcpy`, `strlen`, `printf`, `fopen` |
| Работает с ядром | Да | Нет (или косвенно) |
| Инструкция | `syscall` | `call` |
| Стоимость | Дороже (переключение режимов) | Дешевле |
| Раздел man | 2 (`man 2 write`) | 3 (`man 3 strcpy`) |

**Пример:** `printf` — библиотечная функция. Она форматирует строку в user mode, а затем вызывает `write` (системный вызов) для вывода на терминал.

### Как определить, системный вызов или функция

1. **Логически:** нужны ли ядру эти данные? `strcpy` копирует данные в рамках одного процесса — ядро не нужно. `write` пишет на диск или терминал — ядро обязательно.
2. **По документации:** `man 2 write` — системный вызов; `man 3 strcpy` — библиотечная функция.

### Python и системные вызовы

В Python есть несколько уровней работы с файлами:

```python
# Высокий уровень (много надстроек, буферизация)
with open('file.txt', 'r') as f:
    data = f.read()

# Низкий уровень через модуль os (прямой доступ к системным вызовам)
import os
fd = os.open('file.txt', os.O_RDONLY)
data = os.read(fd, 1024)
os.close(fd)
```

Модуль `os` — «тонкая обёртка» над системными вызовами. В этой части курса мы будем работать именно с ним.

## 5. Создание процессов: системный вызов `fork`

### Жизненный цикл запуска процесса

Запуск одной команды в bash сводится к четырём системным вызовам:
1. **`fork`** — создать новый процесс (дочерний)
2. **`exec`** — загрузить в дочерний процесс новый исполняемый файл
3. **`wait`** — родитель ждёт завершения дочернего процесса
4. **`exit`** — дочерний процесс сообщает о завершении

### Как работает `fork`

`fork` не принимает параметров. При вызове он создаёт **точную копию** текущего процесса: та же память, те же открытые файловые дескрипторы, то же место в коде.

После `fork` существуют **два** процесса, каждый из которых «возвращается» из `fork` — но с разными возвращаемыми значениями:

| Процесс | Возвращаемое значение `fork` |
|---|---|
| **Родитель** | PID дочернего процесса (> 0) |
| **Дочерний** | `0` |
| При ошибке | `-1` (только в родителе, дочерний не создан) |

```python
import os

print(f"Before fork: PID={os.getpid()}")

result = os.fork()

if result == 0:
    print(f"Child:  PID={os.getpid()}, parent PID={os.getppid()}")
else:
    print(f"Parent: PID={os.getpid()}, child PID={result}")
```

Вывод:
```
Before fork: PID=1234
Parent: PID=1234, child PID=1235
Child:  PID=1235, parent PID=1234
```

### Почему ребёнок получает 0, а не свой PID

Каждый процесс в любой момент может узнать свой PID через `os.getpid()`. Родитель же не имеет другого способа узнать PID только что созданного ребёнка — поэтому именно родитель получает PID ребёнка из `fork`.

Родительский PID можно получить через `os.getppid()` (get parent PID).

### Пример: параллельная обработка файлов

```python
import os

result = os.fork()

if result == 0:
    # Дочерний: обрабатывает нечётные файлы
    process_files(odd_files)
else:
    # Родитель: обрабатывает чётные файлы
    process_files(even_files)
```

### Порядок выполнения после fork

После `fork` оба процесса конкурируют за процессорное время на равных правах. **Порядок их выполнения не определён** — нельзя полагаться на то, что родитель или ребёнок напечатает что-то первым. Для упорядочивания нужна явная синхронизация.

## 6. Иерархия процессов

`fork` формирует **дерево процессов**: у каждого процесса (кроме одного корневого) есть ровно один родитель, и произвольное количество детей.

```bash
pstree       # показать дерево процессов
ps -ef       # показать все процессы с PPID (parent PID)
```

Корневой процесс в Linux — `init` (или `systemd`, PID=1). Он создаётся ядром при запуске системы и является предком всех остальных процессов.
