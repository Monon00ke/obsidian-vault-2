## 1. Что такое терминал

### Ограниченность концепции стандартного ввода-вывода

Стандартный вход и стандартный выход — это потоки байтов. Но многие приложения требуют большего:

- **Управляющие клавиши:** стрелки, Backspace, Ctrl+C, Ctrl+Z — это не обычные символы.
- **Позиционирование курсора:** текстовый редактор должен знать ширину и высоту экрана.
- **Цветной вывод:** управляющие последовательности байтов, задающие цвет.
- **Интерактивная строка ввода:** возможность редактировать текст до нажатия Enter.

Всё это — за рамками простого потока байтов. Для этого существует абстракция **терминала**.

### Формы терминала

| Форма | Описание |
|---|---|
| Графический эмулятор | Окно в DE (GNOME Terminal, Konsole и т.д.) |
| Текстовый терминал | Linux без графики (tty1–tty6, Alt+F1…F6) |
| SSH-сессия | Удалённое подключение; рендеринг у клиента, вычисления на сервере |
| Исторический аппаратный терминал | Клавиатура + монитор, подключённые к мейнфрейму |

Графические эмуляторы терминала имитируют исторические аппаратные терминалы, сохраняя совместимость с протоколами управляющих последовательностей (ANSI escape codes, vt100 и др.).

### Терминал как файл

В Linux терминал — тоже файл: устройство в `/dev` (например, `/dev/pts/0`). Запись в этот файл выводит текст на экран; чтение возвращает клавиатурный ввод, включая управляющие клавиши.

Библиотека **ncurses** предоставляет высокоуровневый API для работы с терминалом (позиционирование, цвет, ввод специальных клавиш) — без необходимости вручную формировать управляющие последовательности.

## 2. Управляющий терминал (controlling terminal)

### Связь сессии с терминалом

С каждым **сеансом** (session) связан один **управляющий терминал**. Именно в нём пользователь запустил shell, и именно с ним взаимодействует вся иерархия процессов сеанса.

### Foreground и background группы процессов

В сеансе может работать несколько групп процессов, но доступ к терминальному **вводу** имеет только одна — **foreground-группа**.

| Состояние | Клавиатурный ввод | Вывод |
|---|---|---|
| **Foreground** | Да (эксклюзивно) | Да |
| **Background** | Нет | Да (по умолчанию) |
| **Stopped** | Нет | Нет |

### Переключение между foreground и background

```bash
Ctrl+Z        # остановить текущий foreground-процесс (SIGTSTP)
bg            # продолжить остановленный процесс в background (SIGCONT)
bg %2         # перевести job №2 в background
fg            # перевести последний job обратно в foreground
fg %2         # перевести job №2 в foreground

jobs          # список всех jobs текущего shell
```

### Что происходит при попытке фонового процесса читать stdin

Если фоновый процесс пытается прочитать данные со стандартного входа, привязанного к терминалу, ядро посылает ему сигнал **SIGTTIN**. Дефолтная реакция — процесс засыпает (переходит в stopped). Это объясняет, почему фоновый процесс, ожидающий ввода, всегда висит в состоянии `stopped`, даже после команды `bg`.

**Решение:** для фоновых процессов перенаправлять stdin из файла:

```bash
./my_process < /dev/null &    # отвязать от терминального ввода
./my_process < input.txt &    # читать из файла
```

## 3. Сигнал SIGHUP: что происходит при закрытии терминала

### Почему закрытие терминала убивает процессы

При закрытии терминала (закрытие окна, разрыв SSH-соединения, `logout`) ядро отправляет сигнал **SIGHUP** (signal hang-up) всем группам процессов сеанса. Дефолтная реакция на SIGHUP — завершение процесса. Поэтому все фоновые процессы, запущенные в сеансе, прекращают работу.

**Причины существования такого поведения:**
- Защита от «брошенных» процессов, продолжающих потреблять ресурсы.
- При облачной почасовой аренде — гарантия, что после logout вычисления прекратятся.

### Утилита `nohup`

`nohup` (no hang-up) запускает команду с игнорированием SIGHUP:

```bash
nohup python3 train_model.py > training.log 2>&1 &
# Процесс продолжит работу после закрытия терминала
```

`nohup` устанавливает обработчик `SIG_IGN` на SIGHUP перед запуском команды. Поскольку настройки сигналов наследуются через `exec()`, запущенная программа тоже будет игнорировать SIGHUP.

### Daemon-процессы

Сервисы (базы данных, веб-серверы и т.д.) должны работать 24/7 независимо от терминалов. Для этого они:

1. Делают `fork()` → родитель завершается.
2. В дочернем вызывают `setsid()` → создают новую сессию без управляющего терминала.
3. Делают второй `fork()` → гарантируют, что процесс никогда не станет лидером сессии и не получит управляющий терминал.
4. Перенаправляют stdin/stdout/stderr в `/dev/null` или лог-файлы.

```python
import os, sys

# Первый fork
pid = os.fork()
if pid > 0:
    sys.exit(0)   # родитель завершается

# Создать новую сессию
os.setsid()

# Второй fork (опционально)
pid = os.fork()
if pid > 0:
    sys.exit(0)

# Перенаправить потоки
os.open('/dev/null', os.O_RDONLY)   # stdin
os.open('/dev/null', os.O_WRONLY)   # stdout
os.open('/dev/null', os.O_WRONLY)   # stderr
```

### Другое применение SIGHUP: перечитать конфигурацию

Многие серверные приложения обрабатывают SIGHUP как команду перечитать конфигурационный файл без перезапуска:

```python
import signal

def reload_config(signum, frame):
    global config
    config = load_config('config.yaml')
    print("Config reloaded")

signal.signal(signal.SIGHUP, reload_config)
```

```bash
kill -SIGHUP $(pidof nginx)   # перечитать конфиг nginx
```

## 4. Сигнал SIGCHLD

**SIGCHLD** — сигнал, который ядро посылает родительскому процессу, когда один из его дочерних процессов завершается или останавливается.

Применение: асинхронный сбор статусов завершения дочерних процессов.

```python
import os, signal

def sigchld_handler(signum, frame):
    # Несколько детей могут завершиться почти одновременно,
    # поэтому вызываем waitpid в цикле
    while True:
        try:
            pid, status = os.waitpid(-1, os.WNOHANG)
            if pid == 0:
                break   # нет больше завершившихся детей
            exit_code = status >> 8
            print(f"Child {pid} exited with code {exit_code}")
        except ChildProcessError:
            break

signal.signal(signal.SIGCHLD, sigchld_handler)

# Запустить несколько дочерних процессов
for i in range(5):
    pid = os.fork()
    if pid == 0:
        os._exit(i)   # ребёнок завершается с кодом i

# Родитель продолжает работать
while True:
    pass
```

**Важно:** поскольку сигналы работают по гарантии «at most once», два одновременно завершившихся ребёнка могут породить только один SIGCHLD. Поэтому обработчик должен вызывать `waitpid` в цикле с `WNOHANG`, а не единожды.

## 5. Атомарность и гонки состояния

### Проблема check-then-act

```python
# НЕБЕЗОПАСНО: между проверкой и действием другой процесс может удалить файл
if os.path.exists('data.txt'):
    with open('data.txt') as f:   # может упасть здесь!
        data = f.read()
```

**Решение:** выполнять действие напрямую и обрабатывать ошибки:

```python
try:
    with open('data.txt') as f:
        data = f.read()
except FileNotFoundError:
    # обработать отсутствие файла
```

### Проблема параллельного обновления разделяемого состояния

```
Процесс A:  читает X=10
Процесс B:  читает X=10
Процесс A:  пишет X=11  (X+1)
Процесс B:  пишет X=11  (X+1)  ← должно было быть 12!
```

Два процесса сделали по одному обновлению, но одно обновление потеряно. Решение — **лок** или **семафор**.

## 6. Локи и семафоры

### Лок (Lock / Mutex)

**Лок** — примитив синхронизации, который может захватить ровно один исполнитель. Остальные, пытающиеся захватить занятый лок, блокируются до его освобождения.

```
Операции: acquire() / lock()   — захватить
          release() / unlock() — освободить
```

### Семафор

**Семафор** — обобщение лока. Вместо одного слота — счётчик вместимости N. Каждый `acquire()` уменьшает счётчик на 1; при счётчике 0 — блокировка. Каждый `release()` увеличивает счётчик.

```
Семафор(1) == Лок (ровно один)
Семафор(N) == N параллельных входов
```

**Пример применения:** ограничить число одновременных запросов к одному серверу:

```python
from multiprocessing import Semaphore

sem = Semaphore(3)   # не более 3 одновременных запросов

def download(url):
    with sem:   # acquire при входе, release при выходе
        return requests.get(url)
```

### POSIX семафоры и System V семафоры

Ядро Linux предоставляет два механизма:

| | POSIX семафоры | System V (SysV) семафоры |
|---|---|---|
| Стандарт | Современный POSIX | Исторический AT&T SysV |
| Именование | Через файловую систему (`/dev/shm`) | Числовой IPC-ключ |
| Управление из командной строки | Сложнее | Через `ipcs`/`ipcrm` |

```bash
# Просмотр System V IPC объектов (семафоры, разделяемая память, очереди)
ipcs

# Удалить семафор по ID
ipcrm -s <semid>
```

System V семафоры удобнее наблюдать и управлять из командной строки — они видны через `ipcs` независимо от того, какой процесс их создал.

## 7. Классические проблемы: брошенный лок и дедлок

### Брошенный лок

Если процесс захватил лок, а потом завершился (по исключению, сигналу и т.д.) не освободив его, все ожидающие заблокируются навсегда.

**Профилактика:**

```python
import threading

lock = threading.Lock()

# Всегда используйте контекстный менеджер
with lock:
    do_critical_work()
# lock.release() вызовется автоматически, даже при исключении

# Обработчик сигналов для аккуратного выхода
import signal, sys

def on_signal(signum, frame):
    lock.release()   # освободить перед выходом
    sys.exit(0)

signal.signal(signal.SIGTERM, on_signal)
```

### Дедлок (Deadlock)

Дедлок возникает, когда два или более процессов циклически ожидают друг друга:

```
Процесс A захватил Lock_1, ждёт Lock_2
Процесс B захватил Lock_2, ждёт Lock_1
→ оба ждут вечно
```

В общем случае дедлок — это **цикл в графе зависимостей ожидания**.

**Профилактика — строгий порядок захвата:**

Если все процессы всегда захватывают локи в одном и том же предопределённом порядке, цикл зависимости невозможен.

```python
# Правило: всегда Lock_1 перед Lock_2
with lock_1:
    with lock_2:
        do_work()
# Нельзя делать наоборот: with lock_2, with lock_1 — риск дедлока!
```

**Другие подходы:**
- `try_acquire()` с таймаутом: если за N секунд не получили — отпустить всё и повторить.
- Обнаружение дедлока: периодически анализировать граф ожидания (применяется в СУБД).
