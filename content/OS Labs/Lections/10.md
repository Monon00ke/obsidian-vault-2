## 1. Системный вызов `exec`

### Что делает `exec`

`exec` — системный вызов, который **не запускает новый процесс**, а **перерождает текущий**: заменяет его код, данные и таблицы страниц содержимым нового исполняемого файла. PID процесса при этом остаётся тем же.

```
До exec:  [ Python-интерпретатор + ваш скрипт | PID=1234 ]
После exec: [ /usr/bin/ls                       | PID=1234 ]
```

Код после `exec` в исходном процессе **никогда не выполнится** (в Python будет брошено исключение при ошибке; в C код после `exec` достижим только при ошибке).

### Семейство функций exec

В стандарте POSIX несколько вариантов (`execv`, `execve`, `execl`, `execvp` и др.) — все с одинаковым смыслом, различающиеся способом передачи аргументов и переменных окружения. В Python через `os.execv`:

```python
import os

# Первый аргумент — полный путь к файлу (не короткое имя!)
# Второй аргумент — список argv: argv[0] обычно = имя программы
os.execv('/usr/bin/ls', ['ls', '-l', '/tmp'])
# Код ниже не выполнится никогда
print("Этот print не выполнится")
```

### Важные нюансы

**Первый аргумент — полный путь к файлу.** Поиск по `PATH` — задача шелла, а не `exec`. Системный вызов принимает только конкретный путь к исполняемому файлу.

**Нулевой элемент `argv`** — традиционно имя программы (так, как её вызвали). Именно его процесс видит в `sys.argv[0]` или `argv[0]` в C.

**Что сохраняется после exec:**
- PID процесса
- Файловые дескрипторы (если не помечены `O_CLOEXEC`)
- Некоторые настройки обработки сигналов
- Пользовательский контекст (UID, GID)

**Что сбрасывается:**
- Код, данные, стек, куча (полная замена)
- Все Python-объекты, переменные, состояние интерпретатора

### exec без fork

`exec` может быть полезен и без `fork`. Например: сформировать сложную командную строку в скрипте, настроить обработку сигналов — и затем через `exec` передать управление нужному процессу, не возвращаясь.

### exec в паре с fork (шаблон шелла)

```python
import os, sys

pid = os.fork()

if pid == 0:
    # Дочерний: заменяем себя нужной программой
    try:
        os.execv('/usr/bin/ls', ['ls', '-l'])
    except Exception as e:
        print(f"Failed to run: {e}", file=sys.stderr)
        sys.stderr.flush()
        os._exit(1)   # os._exit, не sys.exit — прямой системный вызов
else:
    # Родитель: ждём завершения ребёнка
    pid, status = os.waitpid(pid, 0)
    print(f"Child exited with status: {status >> 8}")
```

## 2. Системный вызов `exit`

### Особенности exit

`exit` всегда завершается успешно — операционная система не может ему отказать. Причина: завершение процесса не может «зависнуть» в ожидании внешних ресурсов (диска, сети). Если данные в открытых файлах не удалось сбросить — `exit` делает одну попытку и завершается независимо от результата.

При завершении процесса гарантированно освобождается вся память и все структуры ядра, связанные с этим процессом.

### Код завершения

`exit` принимает целочисленный **код завершения**:
- `0` — успешное завершение
- Ненулевое — ошибка (конкретное значение — тип ошибки)

Код завершения передаётся родительскому процессу через `wait`.

### os._exit vs sys.exit в Python

| | `os._exit(n)` | `sys.exit(n)` |
|---|---|---|
| Уровень | Прямой системный вызов | Высокоуровневая Python-функция |
| Закрывает файлы | Нет | Пытается |
| Очищает Python-объекты | Нет | Да (вызывает деструкторы) |
| Сбрасывает буферы | Нет — нужен явный `flush()` | Да |
| Когда использовать | В дочернем процессе после fork | В обычных программах |

**Важно:** в дочернем процессе после `fork` используйте `os._exit()`, а не `sys.exit()`, чтобы избежать двойного выполнения Python-финализаторов.

## 3. Системный вызов `wait`

`wait` переводит вызывающий процесс в состояние ожидания до завершения одного из дочерних процессов. Возвращает PID завершившегося дочернего и его статус.

```python
import os

# Ждать конкретного ребёнка
pid, status = os.waitpid(child_pid, 0)
exit_code = status >> 8  # код завершения — старший байт статуса

# Ждать любого ребёнка
pid, status = os.wait()
```

**Зачем нужен wait:**
- Получить код завершения дочернего процесса.
- Дать ядру окончательно очистить ресурсы завершившегося процесса.
- Синхронизировать выполнение: продолжить работу только после того, как ребёнок закончил.

## 4. Полный цикл: fork → exec → exit → wait

Четыре системных вызова для запуска команды в шелл-стиле:

```
Родитель                    Дочерний
   |                            |
   |--- fork() ---------------→ |  (создать дочерний)
   |                            |
   |                            |--- exec() (загрузить программу)
   |                            |
   |                            |--- ... (работа программы) ...
   |                            |
   |                            |--- exit(code)
   |                            |
   |←-- wait() ----------------/   (получить код завершения)
   |
   | (продолжить работу)
```

### Минимальный шелл на Python

```python
import os, sys

while True:
    # Читаем команду
    try:
        line = input("$ ")
    except EOFError:
        break

    parts = line.strip().split()
    if not parts:
        continue

    # fork
    pid = os.fork()

    if pid == 0:
        # Дочерний: exec
        try:
            os.execv(parts[0], parts)
        except Exception as e:
            print(f"Failed to run '{parts[0]}': {e}", file=sys.stderr)
            sys.stderr.flush()
            os._exit(1)
    else:
        # Родитель: wait
        _, status = os.waitpid(pid, 0)
```

## 5. Состояния процесса

Процесс в ходе жизни переходит между состояниями:

| Состояние | Описание |
|---|---|
| **Running** | Исполняется прямо сейчас на процессоре |
| **Runnable** | Готов к исполнению, ждёт своей очереди на процессор |
| **Waiting (interruptible)** | Ждёт события (ввод-вывод, сигнал, семафор) |
| **Waiting (uninterruptible)** | Ждёт события, которое нельзя прервать (например, дисковый ввод-вывод в ядре) |
| **Zombie** | Завершился, но родитель ещё не забрал код завершения через `wait` |
| **Stopped** | Остановлен сигналом (Ctrl+Z → SIGSTOP) |

```bash
ps aux        # STAT колонка: R=running, S=sleeping, Z=zombie, T=stopped
```

## 6. Зомби-процессы

**Зомби** (zombie) — процесс, который завершился, но чья запись в таблице процессов ещё не удалена, потому что родитель не вызвал `wait`.

Зомби практически не потребляет ресурсов (несколько десятков байт), но занимает PID. Большое количество зомби — признак того, что родительский процесс некорректно обрабатывает завершение детей.

**Как избавиться от зомби:**
```bash
# Найти родителя зомби
ps -el | grep Z     # найти зомби-процессы
ps -p <zombie_pid> -o ppid=   # найти PID родителя

# Убить родителя (дети усыновятся init/systemd, который их корректно "waitнет")
kill <parent_pid>
```

## 7. Иерархия процессов и усыновление

Все процессы образуют **дерево** с единым корнем — процессом с PID=1 (`init` или `systemd`).

Если родительский процесс завершается раньше дочернего:
- Дочерний процесс **усыновляется** процессом PID=1.
- PID=1 никогда не завершается; он всегда готов вызвать `wait` для своих новых «детей».
- Это предотвращает появление «осиротевших» процессов без родителя.

## 8. Межпроцессное взаимодействие (IPC — Inter-Process Communication)

Изолированные процессы могут общаться через несколько механизмов:

| Механизм | Описание | Особенность |
|---|---|---|
| **Pipe** | Однонаправленный буфер в памяти | Только между родственными процессами (после fork) |
| **FIFO (Named pipe)** | Как pipe, но привязан к файловой системе | Между любыми процессами |
| **Разделяемая память** | Общая страница памяти для нескольких процессов | Самый быстрый способ; нужна синхронизация |
| **Семафор** | Счётчик для синхронизации доступа к ресурсу | Ограничивает количество одновременных входов |
| **Очередь сообщений** | Асинхронная передача структурированных сообщений | Удобнее pipe для сложных протоколов |
| **Сигналы** | Асинхронные уведомления процессу | Ограниченная полезная нагрузка (только номер сигнала) |
| **Unix-сокет** | Двунаправленное соединение через файл | Произвольные процессы на одной машине |
| **TCP/UDP-сокет** | Сетевое взаимодействие | Процессы на разных машинах |

### Pipe

Pipe создаётся вызовом `os.pipe()`, который возвращает пару файловых дескрипторов: один для чтения, один для записи.

```python
import os

r_fd, w_fd = os.pipe()   # создать pipe

pid = os.fork()

if pid == 0:
    # Дочерний: пишет в pipe
    os.close(r_fd)               # закрыть ненужный конец
    os.write(w_fd, b"hello\n")
    os.close(w_fd)
    os._exit(0)
else:
    # Родитель: читает из pipe
    os.close(w_fd)               # закрыть ненужный конец
    data = os.read(r_fd, 1024)
    os.close(r_fd)
    print("Received:", data.decode())
    os.wait()
```

Именно через pipe организуются конвейеры (`|`) в шелле: `stdout` левого процесса подключается к `stdin` правого через общий pipe.
