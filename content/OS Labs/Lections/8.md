# Лекция: Процессы, управление памятью и виртуальное адресное пространство

---

## 1. Что такое процесс

**Процесс** — это запущенная в памяти программа. Файл на диске — это просто набор байтов; процесс — это тот же код, загруженный в оперативную память и исполняемый процессором.

Ключевые свойства:
- Один исполняемый файл можно запустить несколько раз — получится несколько процессов.
- Bash в командной строке — тоже процесс.
- Когда вы запускаете `grep`, `sed`, `awk` — вы запускаете новый процесс.

### Параллелизм

Одновременно работающих процессов в типичной Linux-системе значительно больше, чем аппаратных ядер процессора — это нормально. Причина: большинство процессов большую часть времени ждут (ввода-вывода, сигнала, таймера), а не считают. Операционная система переключает ядра между ними.

## 2. Синхронный и фоновый запуск процессов

### Синхронный запуск (обычный)

```bash
du -hs /
```

Bash запускает процесс и **ждёт** его завершения. В это время ввод с клавиатуры недоступен.

### Фоновый запуск (`&`)

```bash
du -hs / 2>/dev/null &
```

Символ `&` запускает процесс в фоне. Bash немедленно возвращает управление пользователю; оба процесса работают параллельно.

**Нюансы фонового запуска:**

| Проблема | Рекомендация |
|---|---|
| stdout фонового процесса смешивается с текущим терминалом | Перенаправлять в файл или `/dev/null` |
| stdin фонового процесса недоступен | Процессы, читающие stdin, подвиснут в ожидании |
| Конвейер запущен фоново | Внутренний stdin/stdout конвейера работают нормально; внешний stdout смешивается с терминалом |

```bash
# Пример: запустить долгий конвейер фоново, сохраняя вывод
seq 1 100000 | wc -l > result.txt 2>/dev/null &
```

### Управление фоновыми процессами

```bash
jobs          # список фоновых задач текущей оболочки
fg            # перевести последнюю задачу на передний план
fg %2         # перевести задачу №2 на передний план
bg            # продолжить остановленную задачу в фоне
Ctrl+Z        # остановить текущий процесс (перевести в suspended)
```

## 3. Из чего состоит процесс

Каждый процесс в памяти содержит:

| Область | Описание | Размер |
|---|---|---|
| **Код** (text) | Машинные инструкции | Статически известен при загрузке |
| **Статические/глобальные данные** | Глобальные и статические переменные | Статически известен при загрузке |
| **Stack (стек вызовов)** | Локальные переменные, адреса возврата | Ограничен, обычно 1–8 МБ |
| **Heap (куча)** | Динамически выделяемая память (`malloc`, `new`) | Растёт по мере работы программы |

Помимо памяти, с процессом связаны открытые **файловые дескрипторы**, переменные окружения, права доступа и другие ресурсы ядра.

### Как это соотносится с Python

Когда вы запускаете Python-скрипт, операционная система запускает процесс **интерпретатора** (`python3`). Ваш `.py`-файл — не исполняемый объект с точки зрения ОС; его читает и исполняет интерпретатор. Создание объектов в Python → вызовы `malloc` внутри интерпретатора → выделение памяти из ОС.

## 4. Стек вызовов и переполнение стека

**Stack вызовов** растёт вниз (от старших адресов к младшим). При каждом вызове функции на стек помещаются:
- адрес возврата;
- локальные переменные функции;
- аргументы.

**Проблема:** размер стека фиксирован и невелик — порядка 1–8 МБ (зависит от ОС и настроек), что соответствует примерно 10 000–50 000 уровней вложенности.

Это имеет практические последствия:

```python
# Обход графа из миллиона вершин рекурсивно
def dfs(v):
    for u in graph[v]:
        dfs(u)  # глубина может достичь миллиона → RecursionError
```

**Решение для глубоких структур:** реализовывать обход итеративно, используя явный стек как структуру данных в куче (куча не имеет жёсткого ограничения на размер).

```python
# Итеративный DFS — стек в куче, без ограничений рекурсии
def dfs_iterative(start):
    stack = [start]
    while stack:
        v = stack.pop()
        for u in graph[v]:
            stack.append(u)
```

## 5. Динамическая память (куча) и её возврат в ОС

### Как работает malloc/free

`malloc` запрашивает у ОС память блоками. `free` возвращает память **менеджеру памяти внутри процесса**, а не сразу в ОС. Менеджер памяти переиспользует освобождённые блоки при следующих вызовах `malloc`.

Возврат памяти в ОС происходит только если можно «сдвинуть» верхнюю границу кучи назад. Если в конце кучи есть хотя бы один занятый блок — память не может быть возвращена в ОС, даже если всё остальное свободно.

**Практическое следствие:** закрытие вкладок в браузере или окон в IDE может не возвращать оперативную память системе немедленно. Перезапуск приложения — гарантированное освобождение всей памяти.

### Гарантии при завершении процесса

| Ресурс | Гарантия при exit() |
|---|---|
| Оперативная память (код, данные, стек, куча) | ✅ Всегда освобождается |
| Файловые дескрипторы (структуры ядра) | ✅ Всегда закрываются |
| Данные записаны на диск | ❌ Не гарантируется |
| Сетевые соединения корректно закрыты | ❌ Не гарантируется |

**Почему данные могут не сохраниться:** `exit()` должен всегда завершаться. Если перед завершением нужно записать данные на диск, а диск недоступен (вытащили флешку, сбой контроллера) — `exit()` не может бесконечно ждать. Поэтому запись на диск при завершении процесса не гарантируется.

**Правило:** если сохранность данных критична — явно вызывайте `flush()` или `fsync()` до завершения программы.

```python
# Python: явный сброс буферов
with open('data.txt', 'w') as f:
    f.write(data)
    f.flush()      # сбросить буфер Python → ОС
    os.fsync(f.fileno())  # сбросить буфер ОС → диск
```

## 6. Виртуальная память и страничная организация

### Зачем нужна виртуализация памяти

Без неё возникают две серьёзные проблемы:

1. **Проблема компиляции.** Компилятор не знает, по каким физическим адресам окажется программа в памяти в момент запуска — рядом работают другие процессы.

2. **Проблема изоляции.** Если все процессы видят одно физическое адресное пространство, ошибочный (или вредоносный) код может читать и перезаписывать данные других процессов.

### Страницы памяти

Физическая память делится на блоки по 4 КБ, называемые **страницами**. Виртуальный адрес, с которым работает программа, состоит из двух частей:

```
64-битный виртуальный адрес:
[ ... номер страницы (52 бит) ... | смещение внутри страницы (12 бит) ]
```

Смещение (12 бит = 4096 вариантов) указывает на конкретный байт внутри страницы и передаётся напрямую. Номер страницы пропускается через **таблицу страниц**.

### Таблица страниц

**Таблица страниц** — структура данных, хранящаяся в памяти. По виртуальному номеру страницы она возвращает физический адрес страницы.

```
Виртуальный адрес → [таблица страниц] → Физический адрес
```

Ключевые свойства:
- У каждого процесса **своя** таблица страниц.
- При переключении контекста процессов ОС меняет указатель на таблицу страниц в специальном регистре процессора.
- Преобразование адресов выполняет **аппаратно** специальный блок процессора — MMU (Memory Management Unit). Никакого дополнительного кода компилятора для этого не нужно.
- MMU имеет кэш последних преобразований (TLB), поэтому накладные расходы минимальны.

### Многоуровневые таблицы страниц

Хранить таблицу страниц в виде плоского массива с $2^{52}$ записями слишком расточительно. На практике используется **четырёхуровневая** схема (в процессорах Intel x86-64):

```
Виртуальный адрес разбивается на 5 частей:
[ L4 (9 бит) | L3 (9 бит) | L2 (9 бит) | L1 (9 бит) | смещение (12 бит) ]

L4-таблица → L3-таблица → L2-таблица → L1-таблица → физический адрес
```

Каждый уровень содержит не более 512 записей (2⁹). Таблицы нижних уровней создаются только для реально используемых диапазонов памяти — большие «дыры» в адресном пространстве места не занимают.

### Как виртуальная память решает основные проблемы

**Проблема компиляции:** Компилятор генерирует код, использующий виртуальные адреса. При загрузке программы ОС создаёт таблицу страниц и отображает виртуальные адреса туда, где физически выделена память. Программа «думает», что работает одна во всей памяти.

**Проблема изоляции:** Каждый процесс видит только свою таблицу страниц. Адрес, не присутствующий в таблице, помечен как невалидный. Обращение к нему вызывает аппаратное прерывание → ОС посылает процессу сигнал `SIGSEGV` (Segmentation Fault) → процесс завершается. Прочитать или записать данные другого процесса невозможно.

**Дополнительные возможности:**

| Возможность | Как реализована |
|---|---|
| Защита кода от записи | Биты прав (R/W/X) в записи таблицы страниц |
| Разделяемая память | Два процесса имеют записи в своих таблицах, указывающие на одну физическую страницу |
| Ленивое выделение памяти | Страница помечена как невыделенная; физическая память выделяется при первом обращении |
| Swap (подкачка) | Страница выгружается на диск; при обращении подгружается обратно |

## 7. Наблюдение за процессами

### Просмотр процессов

```bash
ps aux               # все процессы системы
ps aux | grep python # найти процессы Python
top                  # интерактивный мониторинг (q — выход)
htop                 # улучшенная версия top (если установлен)

# Файловая система /proc
ls /proc/            # каталоги с именами = PID процессов
ls /proc/1234/       # информация о процессе с PID 1234
ls /proc/1234/fd/    # открытые файловые дескрипторы
cat /proc/1234/status  # статус процесса: память, UID и т.д.
```

### Память процесса

```bash
# Объём занятой дисковой памяти
du -hs /path/to/dir

# Использование памяти процессами
ps aux --sort=-%mem | head -20

# Подробная карта памяти конкретного процесса
cat /proc/1234/maps
```

### Завершение процессов

```bash
kill PID             # послать SIGTERM (вежливая просьба завершиться)
kill -9 PID          # послать SIGKILL (принудительное завершение)
killall имя          # завершить все процессы с данным именем
```

## 8. Разделяемая память между процессами

Процессы изолированы, но могут обмениваться данными через:

| Механизм | Описание |
|---|---|
| **Pipe** (`\|`) | Однонаправленный поток байтов между процессами |
| **Файл** | Запись и чтение через файловую систему |
| **Разделяемая память** (`shm`) | Одна физическая страница отображена в оба процесса |
| **Unix-сокет** | Двунаправленное взаимодействие по файлу сокета |
| **Сетевой сокет** | TCP/UDP, в том числе между разными машинами |

## Итог лекции

Разобраны:
- **Процесс**: запущенный в памяти код с данными; отличие от файла на диске;
- **Синхронный и фоновый запуск** (`&`); нюансы stdin/stdout в фоне;
- **Структура памяти процесса**: код, статические данные, стек, куча;
- **Стек вызовов**: ограниченный размер, последствия для рекурсии, итеративные альтернативы;
- **Динамическая память**: `malloc`/`free`, почему память может не возвращаться в ОС немедленно;
- **Гарантии при exit()**: память освобождается всегда, данные на диске — нет;
- **Виртуальная память и страничная организация**: страница 4 КБ, таблица страниц, MMU, четырёхуровневая схема в x86-64; решение проблем компиляции и изоляции;
- **Наблюдение за процессами**: `ps`, `top`, `/proc`.

На следующем занятии: **своп (swap)**, сигналы и работа с процессами.
